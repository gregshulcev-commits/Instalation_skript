CONFIG_GUIDE
============

Как устроены конфиги
--------------------
Каждый интерфейс имеет отдельный серверный конфиг:

  /etc/amnezia/amneziawg/awg0.conf
  /etc/amnezia/amneziawg/awg1.conf

Каждый .conf содержит свой [Interface]:

  PrivateKey
  Address IPv4
  Address IPv6 ULA
  ListenPort
  MTU
  Jc/Jmin/Jmax/S1/S2/H1/H2/H3/H4

Клиенты добавляются ниже в виде блоков:

  ### Client phone
  [Peer]
  PublicKey = ...
  PresharedKey = ...
  AllowedIPs = 10.8.1.2/32

Если CLIENT_ENABLE_IPV6=yes, peer будет содержать:

  AllowedIPs = 10.8.1.2/32, fd42:42:42::2/128

Подсети
-------
Автоматическая выдача IPv4 сейчас рассчитана на /24. Поэтому используйте разные /24 для разных интерфейсов:

  awg0 -> 10.8.1.1/24
  awg1 -> 10.8.2.1/24
  awg2 -> 10.8.3.1/24

Не используйте одну и ту же подсеть на двух интерфейсах.

IPv6 ULA тоже должна отличаться:

  awg0 -> fd42:42:42::1/64
  awg1 -> fd42:42:43::1/64
  awg2 -> fd42:42:44::1/64

MTU
---
В bundle разделены два значения:

1. Server MTU
   Записывается в server .conf:

     MTU = 1280

2. Default client MTU
   Записывается в manager-<iface>.env:

     DEFAULT_CLIENT_MTU='1280'

При добавлении клиента это значение предлагается по умолчанию и попадает в client.conf:

  MTU = 1280

Для проблемных мобильных сетей безопасный первый тест:

  server MTU = 1280
  client MTU = 1280

Если нужно проверить экстремальный вариант:

  server MTU = 1280
  client MTU = 1200

Но MTU ниже 1280 лучше использовать только для IPv4-only клиентов.

AWG параметры
-------------
При создании каждого интерфейса задаются:

  Jc
  Jmin
  Jmax
  S1
  S2
  H1
  H2
  H3
  H4

Эти параметры копируются из server.conf в каждый client.conf при добавлении клиента.

Важно: параметры клиента должны совпадать с параметрами интерфейса сервера. Если вы изменили Jc/Jmin/Jmax/S1/S2/H1-H4 в server.conf вручную, пересоздайте клиентский конфиг или поправьте его вручную.

manager env
-----------
Для каждого интерфейса создаётся:

  /etc/amnezia/amneziawg/manager-awg0.env
  /etc/amnezia/amneziawg/manager-awg1.env

Там лежат:
- SERVER_CONF;
- CLIENTS_DIR;
- KEYS_DIR;
- VPN_IF;
- SERVICE_NAME;
- ENDPOINT_HOST;
- ENDPOINT_PORT;
- DNS_SERVERS;
- DEFAULT_CLIENT_MTU.

Файл manager.env — это только совместимый указатель на последний созданный/изменённый интерфейс. Он обновляется append-only: старые блоки остаются, а shell source использует последние значения. При нескольких интерфейсах лучше всегда указывать имя интерфейса явно:

  sudo ./scripts/04_add_client.sh phone awg1

или пользоваться главным меню:

  sudo ./install.sh

Безопасное добавление интерфейса
--------------------------------
Правильная последовательность:

  sudo ./install.sh
  -> 3) Создать новый интерфейс AWG
  -> задать новую подсеть и порт
  -> согласиться обновить nftables/NAT
  -> при желании добавить клиента

Скрипт не перезаписывает старые .conf. Если указать уже существующее имя интерфейса, операция будет остановлена, даже если задана совместимая переменная OVERWRITE_EXISTING_IFACE=yes. Для изменений старого интерфейса добавляйте клиента через 04_add_client.sh или редактируйте вручную после отдельного backup.

Append-only и backup
--------------------
При добавлении клиента server.conf только дополняется новым [Peer] в конце файла. Перед append создаётся отдельная резервная копия вида:

  file.bak
  file.bak.YYYYMMDD-HHMMSS-PID

Автоматический rollback отключён: восстановление backup поверх server.conf было бы перезаписью старого файла. Если перезапуск сервиса не удался, backup остаётся рядом, а решение об откате принимает администратор вручную.

Установка awg-tools
-------------------
При реальной сборке 01_install_from_source.sh устанавливает awg-tools в новый каталог TOOLS_INSTALL_ROOT. Если базовый каталог уже существует, выбирается свободный путь с суффиксом timestamp/PID. Пути к awg и awg-quick сохраняются в install.env, а systemd unit получает PATH с каталогом установленного awg-quick, чтобы не требовать перезаписи старых системных бинарников.
