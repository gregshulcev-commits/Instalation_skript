TROUBLESHOOTING
===============

Ошибка awg-quick: ip -6 route add ... RTNETLINK answers: No such device
-----------------------------------------------------------------------
Типичный лог:

  awg-quick up awg800
  ip link add awg800 type amneziawg
  awg setconf awg800 /dev/fd/63
  ip -4 address add 10.8.3.1/24 dev awg800
  ip -6 address add fd42:42:44::1/64 dev awg800
  ip link set mtu 800 up dev awg800
  ip -6 route add fd42:42:44::2/128 dev awg800
  RTNETLINK answers: No such device
  ip link delete dev awg800

Что происходит:
- awg-quick видит IPv6 /128 у peer в server.conf AllowedIPs;
- поэтому пытается добавить IPv6 route к peer;
- при некорректной/небезопасной IPv6+MTU конфигурации интерфейс падает.

Что изменено:
- новые клиенты по умолчанию получают в server.conf только IPv4 /32;
- IPv6 /128 добавляется только при CLIENT_ENABLE_IPV6=yes;
- IPv6 + MTU ниже 1280 блокируется.

Что сделать с уже созданным старым конфигом:
1. Откройте server.conf:

     sudo nano /etc/amnezia/amneziawg/awg800.conf

2. Найдите peer, где есть строка вида:

     AllowedIPs = 10.8.3.2/32, fd42:42:44::2/128

3. Если IPv6 не нужен, временно оставьте только IPv4:

     AllowedIPs = 10.8.3.2/32

4. Если IPv6 нужен, поднимите MTU до 1280+ и проверьте внешнюю IPv6-маршрутизацию/firewall.

5. Проверьте:

     sudo awg-quick up /etc/amnezia/amneziawg/awg800.conf

nftables снова показывает дубли
-------------------------------
Запустите пересборку:

  sudo ./scripts/03_setup_nftables.sh

Проверьте, что итоговый файл начинается с:

  flush ruleset

и что AWG-правила представлены одной строкой на группу:

  iifname "ens3" udp dport { 56789, 520 } accept
  iifname { "awg0", "awg1" } oifname "ens3" masquerade

Если после ручных правок runtime ruleset всё равно содержит старые правила, примените полный reload:

  sudo nft -c -f /etc/nftables.conf
  sudo nft -f /etc/nftables.conf

Скрипт отказался менять nftables
--------------------------------
Это ожидаемо, если структура firewall нестандартная или нет понятной пары:

  table inet filter / chain input
  table inet nat / chain postrouting

В этом случае скрипт печатает ручные инструкции и не трогает файл. Сделайте backup вручную, перенесите показанные строки и проверьте `nft -c -f`.

Клиент не подключается извне
----------------------------
Проверьте:

1. Порт слушается в server.conf:

     grep -E 'ListenPort|Address|MTU' /etc/amnezia/amneziawg/awg0.conf

2. nftables разрешает UDP порт на внешнем интерфейсе:

     sudo nft list ruleset | grep -E 'udp dport|masquerade'

3. Endpoint в client.conf указывает на публичный IP/домен и правильный порт.

4. Сервис поднят:

     sudo systemctl status awg-quick@awg0.service --no-pager

5. Логи:

     sudo journalctl -u awg-quick@awg0.service -n 100 --no-pager

Удаление клиента пошло не так
-----------------------------
Каждое удаление создаёт backup:

  /etc/amnezia/amneziawg/backups/YYYYMMDD-HHMMSS-remove-client-IFACE-CLIENT

Откат:

  sudo ./scripts/10_restore_backup.sh /etc/amnezia/amneziawg/backups/YYYYMMDD-HHMMSS-remove-client-IFACE-CLIENT

Удаление интерфейса пошло не так
--------------------------------
Каждое удаление интерфейса создаёт backup:

  /etc/amnezia/amneziawg/backups/YYYYMMDD-HHMMSS-remove-interface-IFACE

Откат:

  sudo RESTORE_CONFIRM=yes RESTORE_APPLY_NFT=yes RESTORE_RESTART_SERVICES=yes \
       ./scripts/10_restore_backup.sh /etc/amnezia/amneziawg/backups/YYYYMMDD-HHMMSS-remove-interface-IFACE

После restore проверьте:

  sudo ./scripts/00_manage.sh --status
  sudo nft -c -f /etc/nftables.conf

Хочу полностью откатить последнюю операцию
-----------------------------------------
Посмотрите последние backup-папки:

  sudo ls -lt /etc/amnezia/amneziawg/backups

Выберите нужную и выполните restore:

  sudo ./scripts/10_restore_backup.sh /etc/amnezia/amneziawg/backups/<папка>

Перед restore скрипт создаст pre-restore backup, так что сам restore тоже можно отменить.
